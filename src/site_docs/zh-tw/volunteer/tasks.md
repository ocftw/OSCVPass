# :material-clipboard-check: 任務指南

本指南將協助你了解如何在 OSCVPass 專案中找到適合的任務、認領並完成它。

## 任務類型

OSCVPass 的任務主要分為以下幾類：

### :material-file-document: 文件類

**適合對象**：喜歡寫作、整理資訊的人

**常見任務**：

- 更新說明文件
- 撰寫 FAQ
- 翻譯或校對內容
- 整理申請範例

**所需技能**：

- 基本中文寫作能力
- Markdown 語法（可以邊做邊學）
- 清楚表達的能力

**時間估計**：1-3 小時 / 任務

**範例任務標籤**：`documentation`、`content`

### :material-palette: 設計類

**適合對象**：有設計背景或美感的人

**常見任務**：

- 設計流程圖、懶人包
- 製作社群宣傳素材
- 優化網站視覺呈現
- 設計簡報範本

**所需技能**：

- 基礎平面設計能力
- 熟悉 Figma、Canva 或其他設計工具
- 願意接受回饋並修改

**時間估計**：2-5 小時 / 任務

**範例任務標籤**：`design`、`visual`

### :material-code-tags: 技術類

**適合對象**：熟悉程式或網站技術的人

**常見任務**：

- 更新 MkDocs 設定
- 執行專案詳細資料更新腳本
- 修正網站 bug
- 優化自動化流程

**所需技能**：

- 基本 Git 操作
- Python、Markdown 或網頁技術
- 願意查文件與除錯

**時間估計**：1-4 小時 / 任務

**範例任務標籤**：`technical`、`automation`

### :material-account-group: 社群類

**適合對象**：喜歡與人互動、經營社群的人

**常見任務**：

- 回覆 Discussions 問題
- 歡迎新志工
- 整理會議紀錄
- 協調志工任務

**所需技能**：

- 良好溝通能力
- 耐心與同理心
- 基本文字組織能力

**時間估計**：1-2 小時 / 任務

**範例任務標籤**：`community`、`communication`

### :material-briefcase: 專案管理類

**適合對象**：喜歡規劃、追蹤進度的人

**常見任務**：

- 更新 GitHub Projects 看板
- 追蹤任務進度
- 整理待辦事項
- 協助拆解大型任務

**所需技能**：

- 組織與規劃能力
- 熟悉專案管理工具
- 主動追蹤與提醒

**時間估計**：1-3 小時 / 任務

**範例任務標籤**：`project-management`、`planning`

## 任務難度分級

### 🟢 新手友善（Good First Issue）

- **特徵**：步驟清楚、影響範圍小、有明確指引
- **範例**：修正文件錯字、更新連結、新增簡單內容
- **建議**：新加入的志工從這裡開始

### 🟡 中等難度

- **特徵**：需要一定背景知識、可能需要討論
- **範例**：撰寫新章節、設計複雜圖表、執行技術腳本
- **建議**：完成 2-3 個新手任務後嘗試

### 🔴 進階任務

- **特徵**：需要深入了解專案、影響範圍大、需要協調
- **範例**：重構文件架構、規劃新功能、建立自動化流程
- **建議**：參與專案 1-2 個月後嘗試

## 如何找到適合的任務

### 方法一：瀏覽 GitHub Issues

1. 前往 [OSCVPass Issues](https://github.com/ocftw/OSCVPass/issues){target="_blank"}
2. 使用標籤篩選：
   - `good first issue`：新手任務
   - `help wanted`：需要協助的任務
   - `documentation`、`design`、`technical` 等：依類型篩選
3. 點開 Issue 閱讀詳細說明
4. 評估是否適合自己

### 方法二：查看 GitHub Projects

1. 前往 [OSCVPass Projects](https://github.com/orgs/ocftw/projects/3){target="_blank"}
2. 查看「待辦」(To Do) 欄位
3. 找到感興趣且沒有 assignee 的任務
4. 點開連結到對應的 Issue

### 方法三：在 Discussions 詢問

如果不確定該做什麼：

1. 前往 [GitHub Discussions](https://github.com/ocftw/OSCVPass/discussions){target="_blank"}
2. 發文說明你的專長與可用時間
3. 其他志工或負責人會推薦適合的任務

## 評估任務是否適合你

在認領任務前，問自己：

- [ ] **我了解這個任務在做什麼嗎？**（不懂可以先問）
- [ ] **我有相關的技能或經驗嗎？**（沒有也可以學習）
- [ ] **我在未來 1-2 週有時間完成嗎？**（評估時間投入）
- [ ] **我知道如何開始嗎？**（Issue 中有說明或可以問）
- [ ] **我有興趣做這件事嗎？**（興趣是最好的動力）

!!! tip "新手建議"
    - 第一個任務選簡單的，建立信心
    - 不確定時，先問再認領
    - 寧可做小任務多次，也不要認領後做不完

## 認領任務流程

### 步驟 1：確認任務可認領

檢查 Issue 是否：

- ✅ 沒有被別人認領（assignee 欄位為空）
- ✅ 沒有標記為 `on hold`（暫緩）或 `blocked`（阻塞）
- ✅ 狀態是 `open`（開放中）

### 步驟 2：在 Issue 中留言

在 Issue 中留言：

```markdown
我想認領這個任務！

預計完成時間：[例如：本週末、下週三]

（如有疑問可以一起提出）
```

### 步驟 3：等待確認

- 通常會在 1 天內確認
- 你會被設定為 assignee
- Issue 狀態會更新為 `in progress`

### 步驟 4：開始執行

- 參考 Issue 中的說明開始工作
- 如有問題隨時在 Issue 中提問
- 建議定期更新進度

## 執行任務的最佳實踐

### 📋 任務開始前

1. **仔細閱讀 Issue 說明**：確保你完全理解要做什麼
2. **查看相關資源**：Issue 中可能有參考連結或範例
3. **準備工作環境**：
    - 文件類：確認 Markdown 編輯器可用
    - 設計類：確認設計工具已安裝
    - 技術類：確認開發環境已設定（參考 [CLAUDE.md](../../CLAUDE.md)）
4. **提出問題**：不清楚的地方先問，不要猜測

### ⚙️ 任務執行中

1. **遵循現有風格**：
    - 文件：參考其他頁面的寫作風格
    - 設計：保持視覺風格一致
    - 技術：遵循現有的程式碼規範
2. **小步前進**：將大任務拆成小步驟，逐步完成
3. **定期儲存**：避免遺失工作成果
4. **更新進度**：在 Issue 中留言告知進度（例如：「已完成 50%」）
5. **遇到困難**：
    - 先嘗試自己解決（查文件、搜尋）
    - 卡住超過 30 分鐘，在 Issue 中求助
    - 說明你已經試過什麼方法

### ✅ 任務完成後

1. **自我檢查**：
    - 文件：檢查錯字、格式、連結是否正常
    - 設計：確認檔案格式、解析度符合需求
    - 技術：測試功能是否正常運作
2. **提交 Pull Request**：
    - 清楚說明你做了什麼
    - 附上截圖或範例（如適用）
    - 連結相關 Issue（`Fixes #123`）
3. **回應 Review**：
    - 感謝 reviewer 的回饋
    - 根據建議進行修改
    - 不懂的地方可以討論
4. **慶祝完成**：恭喜你完成一個任務！🎉

## 常見情況處理

### 情況 1：我認領了但做不完

**不要慌張！這很正常。**

處理方式：

1. 在 Issue 中留言說明情況
2. 說明目前進度（完成了什麼、卡在哪裡）
3. 詢問是否有人可以接手，或延長時間

範例留言：

```markdown
抱歉，我發現這個任務比預期複雜，目前我完成了 XX 部分，
但 YY 部分遇到困難。

我可以選擇：
1. 延長 3 天完成（需要協助的部分是...）
2. 將任務交給其他人

請問哪種方式比較好？
```

### 情況 2：我想做但不確定怎麼開始

**先問就對了！**

處理方式：

1. 在 Issue 中留言說明你的困惑
2. 詢問是否有參考資料或範例
3. 請求更詳細的說明

範例留言：

```markdown
我對這個任務有興趣，但不太確定從哪裡開始。

請問：
1. 有沒有類似的範例可以參考？
2. XX 部分應該怎麼做？
3. 預期的成果格式是什麼？

謝謝！
```

### 情況 3：我做完了但不確定對不對

**提交 Pull Request 讓大家幫你看！**

處理方式：

1. 提交 Pull Request（即使不確定）
2. 在 PR 描述中說明你的疑慮
3. 標記為 `Draft`（草稿）或請求 review

範例 PR 描述：

```markdown
我完成了 XX 任務（Fixes #123），但不太確定以下部分：

- 疑問 1：YY 這樣寫是否恰當？
- 疑問 2：格式是否符合規範？

請幫我 review，謝謝！
```

### 情況 4：我想改進 Issue 中的做法

**很好！提出你的想法。**

處理方式：

1. 在 Issue 中留言說明你的想法
2. 解釋為什麼你的方法更好
3. 詢問大家的意見
4. 達成共識後再執行

### 情況 5：任務太大，我只能做一部分

**可以！部分貢獻也很有價值。**

處理方式：

1. 在 Issue 中說明你可以完成哪部分
2. 建議將大任務拆成多個小 Issues
3. 認領其中一個小部分
4. 完成後其他人可以接續

## 時間管理建議

### 新手任務時間參考

| 任務類型 | 預估時間 | 實際建議 |
|---------|---------|---------|
| 修正錯字、更新連結 | 15-30 分鐘 | 新手可能需要 1 小時（含熟悉流程） |
| 撰寫新段落 | 1-2 小時 | 新手可能需要 2-3 小時 |
| 設計簡單圖表 | 2-3 小時 | 含修改可能需要 4-5 小時 |
| 執行技術腳本 | 30 分鐘 - 1 小時 | 新手可能需要 2-3 小時（含環境設定） |

!!! warning "提醒"
    第一次執行任務通常需要更多時間，因為要熟悉流程和工具。這是正常的！

### 時間分配建議

**每週 1 小時的志工**：

- 選擇 1 個小任務（修正錯字、更新連結）
- 或每 2 週完成 1 個中型任務

**每週 2-3 小時的志工**：

- 每週 1 個中型任務
- 或每 2 週 1 個大型任務

**彈性參與**：

- 有空時多做，忙碌時暫停
- 沒有強制要求
